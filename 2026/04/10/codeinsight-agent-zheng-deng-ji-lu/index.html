我的第一个 Agent 项目：CodeInsight-Agent 折腾记录

发表于2026-04-10|更新于2026-04-18|分享
|浏览量:

一、背景

做 CodeInsight-Agent 之前，我对“Agent”这件事的理解其实很模糊。

一开始我只想做一个能和当前仓库互动的本地代码助手，后来才慢慢意识到，真正难的不是“让模型说话”，而是让它在合适的场景里做合适的事。它要知道什么时候只是回答问题，什么时候需要看工作区文件，什么时候应该进入真正的执行模式，甚至还要考虑安全、写文件、跑测试、结果回显这些细节。

这个项目就是沿着这个方向一点点搭起来的。它不是一个炫技 demo，更像是我第一次认真把“AI 能力”和“工程系统”拼在一起的尝试。

从提交记录看，项目大致经历了几个阶段：

  1. 先把基础 Web 代理和 README 搭起来。
  2. 再把消息流、会话管理和页面体验补齐。
  3. 接着引入 workspace_qa 和 agentic 这类不同工作模式。
  4. 然后不断修正路由判断、只读场景、安全边界和检索体验。
  5. 最后再回头做 README、favicon、前端细节和整体结构整理。

这条迭代线很像真实开发，不是先画完蓝图再一次性完成，而是边做边改，边试边补。

二、我在做什么

CodeInsight-Agent 本质上是一个基于 FastAPI 的本地代码助手 Web 应用，主要围绕当前仓库提供三种能力：

  1. QA：纯问答，不读取本地仓库，也不改文件。
  2. workspace_qa：读取当前工作区文件做解释和总结，默认只读。
  3. agentic：以任务板方式执行检索、分析、修改、测试等动作。

它做的事情不算少，但核心目标很明确，就是把“对仓库的理解”和“对仓库的操作”分层。

我实际搭出来的功能主要包括：

  1. 会话管理：支持新建、查看、重命名、置顶、归档和删除。
  2. 工作区浏览：可以看目录树、读文件，在允许写入时还能保存文件。
  3. 模式切换：根据用户输入自动判断应该走哪种模式。
  4. Agent 执行：支持检索、文件读取、目录浏览、补丁写入、命令执行和测试工具。
  5. 流式输出：前端通过 SSE 实时看到中间过程，而不是等最后一个结果。
  6. 评测查看：可以读取 outputs/eval_result.json 的最新评测结果。
  7. 本地路径选择：支持通过系统文件选择器选文件或文件夹。

如果只看表面，它像是一个本地版“代码问答 + 执行助手”。
但真正让我在意的，其实是它背后的路由和权限控制：不是所有请求都该进入 Agent，也不是所有问题都值得读仓库，更不是所有读仓库的请求都应该改文件。

三、关键细节 / 思路

这个项目最花时间的地方，不是某个单独页面，而是“判断”和“边界”。

1. 模式判断比我想象中难

一开始我以为，用户问代码，就去读工作区；用户要修改，就让 Agent 动手。
实际做下来才发现，真实输入非常模糊。

比如同样是“帮我看看这个项目”，有时只是想要概览，有时其实想让系统分析目录结构，有时又隐含了“顺手改一下”的意思。

所以我在 TurnModeDecider 上反复补 marker 和推断逻辑，让系统能更稳地把请求分到：

  1. 纯问答。
  2. 只读工作区分析。
  3. 真正的执行模式。

这部分提交很多，说明一个很现实的事实：Agent 系统里，最贵的往往不是生成答案，而是错误地进入了不该进入的工作流。

2. 只读场景要足够克制

提交记录里有一串很密的修正，几乎都在处理只读请求：

  1. 把“只读的项目分析”路由到 workspace_qa。
  2. 把“文件解释”路由到只读流程。
  3. 把“存在性检查”路由到工作区问答。
  4. 避免只读情况下生成多余的任务板前缀。
  5. 减少只读回合里的工具抖动。

这些改动让我第一次比较具体地理解到，Agent 不只是“能不能做”，还要控制“做多少”。
如果一个只想看代码的人，被系统带着执行了一堆不必要的工具调用，那体验会非常差，甚至比没有 Agent 更差。

3. RAG 和索引不是装饰品

我后来又加了检索评分、评测指标，并且排除了生成产物对索引的干扰。

这件事让我更清楚地认识到，RAG 的问题不只是“有没有检索”，而是：

  1. 检索到的东西是不是对。
  2. 检索结果能不能稳定支持推理。
  3. 索引里有没有脏数据。
  4. 评测能不能反过来告诉我哪里坏了。

如果没有这些约束，检索模块很容易变成“看起来很智能，实际很飘”的东西。

4. 安全和可维护性必须提前考虑

我在提交里做过一些比较实在的收口，比如：

  1. 强化运行时校验。
  2. 约束命令执行安全。
  3. 统一 Web 流和 API 常量。
  4. 释放旧的流锁，避免页面卡住。
  5. 让 JSON 和流响应显式声明 utf-8。

这些看起来不像“Agent 最酷的部分”，但它们决定了这个项目是不是能一直跑下去。

这也是我从第一个 Agent 项目里学到的一个重要经验：
如果一个系统不能稳定地运行，它再聪明也没有意义。

四、总结

做完 CodeInsight-Agent 之后，我对几个东西的理解明显更深了。

第一，是 FastAPI 和流式接口的工程化用法。
我不再只是把它当成一个“写接口的框架”，而是把它当成承载对话、状态和任务流的基础设施。

第二，是 Agent 的分层设计。
QA、workspace_qa、agentic 这三个模式，让我真正体会到：好的 Agent 不是一个大而全的黑盒，而是多个能力边界清晰、职责分明的工作流。

第三，是 RAG 和评测的价值。
检索不是加上去就行，索引、过滤、评分和评测缺一块，整个系统的可靠性都会掉下来。

第四，是我对“自动化”这件事的认识。
我以前会下意识觉得，能自动做很多事就是进步。现在我更倾向于认为，自动化真正的价值在于减少重复劳动，同时不要把不可控性一起放大。

当然，这个项目也有不少不足。

  1. 它还是我第一个 Agent 项目，整体架构是边做边长出来的，不是从一开始就设计得很完整。
  2. 任务路由和意图判断还不够优雅，很多地方靠不断补规则和 marker 在兜底。
  3. 评测体系还可以更系统，现阶段更多是“有了”，离“足够强”还有距离。
  4. 前端和交互体验虽然已经能用，但离成熟产品还有差距。
  5. 对复杂仓库的泛化能力还需要继续打磨，尤其是多轮任务和长上下文场景。

但我觉得这正是第一个 Agent 项目最有价值的地方。
它不一定要一开始就完美，重要的是让我真的走过一遍完整链路：从想法、实现、调试、修正，到最后能清楚地说出它做了什么、为什么这样做、哪里还不够。

如果以后我再做第二个、第三个 Agent 项目，我大概率会更早地把模式边界、评测和安全收口设计进去，而不是等问题堆出来再补。

而这一次的折腾，至少已经把我从“听说过 Agent”带到了“真的做过一个 Agent”。
