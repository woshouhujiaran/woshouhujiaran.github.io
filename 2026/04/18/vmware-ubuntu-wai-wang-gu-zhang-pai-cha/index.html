VMware Ubuntu 虚拟机无法访问外网排查记录：从网关不通到 DNS 恢复

发表于2026-04-18|更新于2026-04-18|分享
|浏览量:

一、背景

这次问题出现在 VMware 里的 Ubuntu 虚拟机上，现象很典型：

  1. ping 不通网关。
  2. ping 8.8.8.8 也失败。
  3. 执行 apt update 报错 Temporary failure resolving。

一开始我以为只是 DNS 配置写坏了，但实际排下来发现是“虚拟网络连通 + 解析链路”一起异常，需要按顺序把链路逐段验证。

二、我在做什么

我按“先网络、后解析”的顺序排查，先确认虚拟机本身有没有拿到基础网络配置：

结果显示网卡有 IP、路由也在，说明不是“网卡没起来”这种低层问题。

然后我继续测虚拟机和 VMware NAT 网关的连通性：

这里直接不通，说明问题不只是在应用层，至少在“虚拟机 -> NAT 网关”这段链路就已经异常。随后再看 apt update 的报错，能确认 DNS 解析链路也处于不可用状态。

三、关键细节 / 思路

这次真正有效的处理是把 DNS 配置重新拉回稳定状态，并让相关服务完整重载。

1. 修复 DNS 配置

我用了两种常见方式之一（按系统环境选其一）：

  1. 直接修改 /etc/resolv.conf，写入公共 DNS。
  2. 通过 systemd-resolved 配置 DNS。

公共 DNS 使用的是：

  • 8.8.8.8
  • 1.1.1.1

2. 重启解析与网络服务

配置改完后，必须让服务重新加载：

然后我重启了虚拟机，确保虚拟网卡、路由和解析配置都按新状态生效。

3. 验证恢复结果

恢复后按顺序验证：

  • ping IP 正常，说明外网链路恢复。
  • ping 域名 正常，说明 DNS 恢复。
  • apt update 不再报 Temporary failure resolving。

四、总结

这次排查给我的结论很明确：遇到虚拟机“不能上网”时，不要一上来只盯 DNS，也不要只看 IP 有没有拿到。

更稳妥的做法是按链路拆开：

  1. ip addr / ip route 先确认本机配置。
  2. ping 网关 判断虚拟机到 NAT 网络是否可达。
  3. ping 外网IP 和 ping 域名 分离“链路问题”和“解析问题”。
  4. 修复 DNS 后重启 systemd-resolved、NetworkManager，必要时重启虚拟机。

这套顺序的好处是：每一步都有明确结论，不会在“猜配置”里来回折腾。
